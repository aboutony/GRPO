2. User-Defined Tables (UDTs)

The GRPO module requires 2 custom tables to be created in SAP B1. These are provisioned automatically by the schema sync script, or can be created manually via Tools → Customization Tools → User-Defined Tables — Setup.

2.1 @GRPO_AUDIT_LOG

Stores the SHA-256 hash chain for regulatory audit compliance.

Column Type Size Description
Code Alphanumeric 50 Primary key (auto-generated)
Name Alphanumeric 100 Display name
U_Action Alphanumeric 50 Action type (GRPO_CREATED, QC_HELD, etc.)
U_DocEntry Numeric 11 Related SAP document entry
U_Operator Alphanumeric 50 Operator who performed action
U_Hash Alphanumeric 64 SHA-256 hash of record
U_PrevHash Alphanumeric 64 Previous record hash (chain link)
U_Payload Memo — Full JSON payload
U_Timestamp Alphanumeric 30 ISO 8601 timestamp

2.2 @GRPO_QC_RECORDS

Stores quality control inspection records and defect evidence.

Column Type Size Description
Code Alphanumeric 50 Primary key
Name Alphanumeric 100 Display name
U_DocEntry Numeric 11 GRPO document entry
U_ItemCode Alphanumeric 50 Item code inspected
U_BatchNum Alphanumeric 36 Batch/lot number
U_DefectCodes Alphanumeric 254 Comma-separated defect codes
U_PhotoPath Alphanumeric 254 Evidence photo file path
U_Inspector Alphanumeric 50 QC inspector name
U_Result Alphanumeric 20 PASS / FAIL / HOLD
U_Notes Memo — Inspection notes
ℹ️ Automated Provisioning
Run npm run schema:sync from the GRPO service to automatically create these tables. The script is idempotent — it will skip tables that already exist.

U_GRPO_QcStatus Overall QC status Alphanumeric 20 Yes
U_GRPO_SaberCertId SABER certificate ID Alphanumeric 50 No
U_GRPO_SaberStatus SABER validation result Alphanumeric 20 No
U_GRPO_AuditHash Document audit hash Alphanumeric 64 No
U_GRPO_DeviceId Mobile device ID Alphanumeric 30 No
U_GRPO_SyncTs Sync timestamp Alphanumeric 30 No

3.2 Line-Level UDFs (PDN1 Table)

Field Name Description Type Size Mandatory
U_GRPO_UdiDi UDI Device Identifier Alphanumeric 50 No
U_GRPO_UdiPi UDI Production Identifier Alphanumeric 100 No
U_GRPO_Sterility Sterility class (S/N/T) Alphanumeric 1 No
U_GRPO_GtinScanned Scanned GTIN-14 barcode Alphanumeric 14 No
U_GRPO_QcReq QC inspection required (Y/N) Alphanumeric 1 No
🔴 Critical — Item Master Setup
For PO-driven receiving to work, each Item Master must have:
• Manage Item by Batches = Yes (for batch-managed medical devices)
• The GTIN must be stored in the Item's Bar Codes table (OBCD)
• If QC inspection is required, set U_GRPO_QcReq = 'Y' on the item

4. DI API & Service Layer Configuration

4.1 Service Layer Setup

The GRPO module uses the SAP B1 Service Layer for PO retrieval and the DI API (COM) for document posting.

1
Verify Service Layer is running
Navigate to https://your-sap-server:50000/b1s/v1/ in a browser.
You should see: {"odata.metadata":"..."}
2
Create a dedicated Service Layer user
In SAP B1: Administration → Setup → General → Users.
Create user GRPO_SERVICE with Super User permissions and a strong password.
3
Test authentication 
POST https://your-sap-server:50000/b1s/v1/Login
Content-Type: application/json

{
  "CompanyDB": "YOUR_COMPANY_DB",
  "UserName": "GRPO_SERVICE",
  "Password": "your-strong-password"
}
Expected: 200 OK with session ID.

4.2 DI API Connection Pool

The GRPO Adapter maintains a connection pool to the DI API for atomic GRPO posting.

Setting Value Description
SAP_SERVER your-sap-server SAP B1 server hostname
SAP_COMPANY_DB YOUR_COMPANY_DB Company database name
SAP_DB_TYPE dst_HANADB dst_HANADB or dst_MSSQL2019
SAP_USER GRPO_SERVICE DI API user
SAP_PASSWORD (encrypted) Stored in environment variable
SAP_LICENSE_SERVER sap-lic:40000 License server address
DI_POOL_SIZE 5 Max concurrent DI API connections

5. GRPO Service Installation

5.1 Environment File

Create a .env file on the GRPO service server:

# SAP B1 Connection
SAP_SERVER=sap-b1.your-domain.local
SAP_COMPANY_DB=YOUR_COMPANY_DB
SAP_DB_TYPE=dst_HANADB
SAP_USER=GRPO_SERVICE
SAP_PASSWORD=your-encrypted-password
SAP_LICENSE_SERVER=sap-lic.your-domain.local:40000

# Service Layer
SERVICE_LAYER_URL=https://sap-b1.your-domain.local:50000/b1s/v1

# GRPO Service
GRPO_PORT=8443
GRPO_HOST=0.0.0.0
SYNC_INTERVAL_MS=3000
MAX_BATCH_SIZE=50
OFFLINE_QUEUE_LIMIT=500

# Regulatory
SABER_API_URL=https://api.saber.sa/v1
SABER_API_KEY=your-saber-api-key
SABER_MODE=strict
SFDA_EXPORT_CRON=0 2 * * *
SFDA_EXPORT_FORMAT=XML

# Security
JWT_SECRET=your-256-bit-secret
AUDIT_HASH_ALGORITHM=SHA-256

1
Run the schema sync

This will create 2 UDTs and 13 UDFs idempotently. Output:

7. Purchase Order Configuration

🔴 Critical — PO-Driven Requirement
The GRPO module operates in PO-Driven mode ONLY. Random product scanning is disabled. Every GRPO must reference a source Purchase Order via BaseType=22, BaseEntry, and BaseLine.

7.1 Purchase Order Requirements

  • PO must have status "Open" to appear in the mobile app
  • All line items must have valid Item Codes that exist in the Item Master
  • Each item must have a GTIN barcode registered in SAP B1 (OBCD table)
  • Batch-managed items must have "Manage Item by Batches" = Yes

7.2 How It Works

1
Operator selects a PO from the mobile PO Selection Screen. Only Open POs are shown.
2
Scanner unlocks. The operator scans each carton's GS1-128 barcode.
3
Strict matching: The scanned GTIN is validated against the PO's line items. If the item is not on the PO → HARD BLOCK with message: "Item not on this PO — Reject Delivery".
4
GRPO posted with BaseEntry/BaseLine referencing the source PO. SAP B1 automatically reduces the PO's open quantities.

8. Barcode Label Requirements

8.1 Supported Formats

Format AI Codes Parsed Priority
GS1-128 (01) GTIN, (10) Batch, (17) Expiry, (21) Serial Primary
GS1 DataMatrix Same as above Primary
QR Code (GS1) Same as above Secondary

8.2 Example GS1-128 Barcode Content

(01)08806555019001(10)BT-2026-0301(17)280615(21)SRL-0001

Breakdown:

AI Field Value
(01) GTIN-14 08806555019001
(10) Batch/Lot BT-2026-0301
(17) Expiry (YYMMDD) 280615 → 2028-06-15
(21) Serial Number SRL-0001
ℹ️ Vendor Label Requirement
Instruct your suppliers to print GS1-128 or GS1 DataMatrix barcodes on all carton labels. The barcode must contain at minimum AI (01) GTIN and AI (10) Batch. Expiry and Serial are recommended for full UDI compliance.

9. SABER API Integration

The GRPO module validates every inbound shipment against the SABER Platform (Saudi Standards, Metrology and Quality Organization) before allowing a GRPO to be posted.

9.1 Configuration

Setting Value
SABER_API_URL https://api.saber.sa/v1
SABER_API_KEY Obtain from saber.sa portal
SABER_MODE strict (blocks invalid) or warn (logs only)

9.2 Validation Flow

  • Valid Certificate → Green badge → GRPO proceeds
  • Expired Certificate → Red block → GRPO blocked, procurement team alerted
  • Not Found → Yellow warning → GRPO held for manual review
✅ For pilot testing
Set SABER_MODE=warn during the pilot phase to log validation results without blocking. Switch to strict for production.

10. SFDA Reporting Configuration

The module generates daily compliance exports for the Saudi Food & Drug Authority (SFDA) Saudi Device Identification (Saudi-DI) database.

Setting Value Description
SFDA_EXPORT_CRON 0 2 * * * Daily at 2:00 AM
SFDA_EXPORT_FORMAT XML XML or CSV
SFDA_EXPORT_PATH /exports/sfda/ Local export directory
SFDA_IDEMPOTENT true Prevents duplicate transmissions

  • Full SAP B1 database backup taken

  • UDFs created and verified (8 header + 5 line)

  • Service Layer connectivity confirmed

  • Item Masters updated: batch management enabled, GTINs registered
  • Test POs created with valid item lines
  • Mobile app installed on all pilot devices
  • Bluetooth scanners paired and tested
  • SABER API key configured (warn mode for pilot)
  • All operators trained on PO-driven receiving flow

Go-Live Day (Day 0)

  • Switch GRPO service to production mode
  • Verify first real PO appears in mobile app
  • Complete first live scan-to-putaway cycle
  • Verify GRPO appears in SAP B1 with correct BaseEntry/BaseLine
  • Confirm PO open quantities reduced correctly
  • Check audit log entry created with valid hash chain
  • Test QC hold routing (if applicable)
  • Notify operations team: "GRPO Module is LIVE"

Post-Go-Live (Day 1–30 Hypercare)

  • Monitor sync latency daily (target: < 5 seconds)
  • Review daily SFDA export completeness
  • Check for failed syncs or connection drops
  • Conduct mock recall drill within first 7 days
  • Achieve 90% operator adoption by day 7
  • Switch SABER mode from warn to strict after day 14
  • Hand over hash chain verifier to QA/Compliance team by day 30

13. Troubleshooting & Support

Issue Cause Resolution
POs not appearing in app PO status is "Closed" or service can't reach Service Layer Verify PO status in SAP B1. Check SERVICE_LAYER_URL connectivity.
Scan returns "Item not on this PO" Item's GTIN not registered in SAP B1 barcodes table Add GTIN to item via Inventory → Item Master Data → Bar Codes tab
GRPO post fails with "Base document error" PO line already fully received, or PO was closed Check PO's remaining open quantities in SAP B1
Sync timeout (> 5s) Network latency or DI API pool exhausted Check network route. Increase DI_POOL_SIZE if needed.
UDF not found error Schema sync was not run on this database Run npm run schema:sync and verify output
SABER validation fails API key expired or certificate not found Verify SABER portal credentials. Check certificate status.
Scanner not connecting Bluetooth HID pairing issue Reset scanner, re-pair. Ensure only one device paired at a time.
Offline scans not syncing Device reconnected but sync queue stalled Force sync via app menu. Check GRPO service logs.
